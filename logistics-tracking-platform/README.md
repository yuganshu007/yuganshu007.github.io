# Logistics Tracking Platform

An event-driven shipment-tracking system built to **back every claim** in these two résumé bullets with runnable code and a reproducible benchmark:

> • Architected **Python/Java microservices** for logistics; optimized **async handlers** and **background workers**; improved end-to-end **response times 40%** under peak load via **tuned request routing** on Linux servers.
>
> • Orchestrated **event-driven tracking pipeline** with **REST APIs** and **message queues** with **retry and validation logic**, **reducing SLA breaches 20%**; applied **SQLite-backed local caching** with **schema indexing** to **cut read latency 40%**.

Everything here runs locally on Linux. The benchmark suite measures each headline number on real traffic so you can reproduce — and defend — them live.

## Architecture

```
                 REST (async handlers)            Redis Streams              background worker pool
   client  ───────────────────────────►  Ingest Service  ──(message queue)──►   Tracking Worker
                                          (Python/FastAPI)                       (Java, N threads)
                                               │  ▲                                   │
                                  GET (cache + │  │ validation                        │ validate + retry + DLQ
                                   index reads)│  │                                   ▼
                                               ▼  └───────────  SQLite (WAL)  ◄────────┘
                                          in-memory LRU+TTL cache        system of record
```

- **Ingest Service** — Python / FastAPI (`ingest-service/`). Async REST handlers. In the optimized
  architecture the hot write path only **validates + routes + publishes** to the message queue and
  returns immediately; reads are served from SQLite behind an in-memory LRU+TTL cache.
- **Request router** (`ingest-service/app/router.py`) — tuned request routing: urgent events
  (`EXCEPTION`/`DELIVERED`) are routed in O(1) to a **high-priority lane** that workers drain first,
  so urgent traffic is not stuck behind a backlog of routine scans under peak load.
- **Tracking Worker** — Java 21 (`tracking-worker/`). A pool of consumer threads reads from a Redis
  Streams **consumer group**, validates each event, calls the (simulated) downstream, **retries**
  transient failures with backoff, routes poison messages to a **dead-letter queue**, and persists
  results to SQLite.
- **Message queue** — Redis Streams with consumer groups (at-least-once delivery, acks, pending list).
- **Store** — a single SQLite database in WAL mode, shared concurrently by the Python reader and the
  Java writers, with secondary **indexes** on the hot query paths.

## REST API

| method | path | purpose |
| --- | --- | --- |
| `POST` | `/shipments` | create a shipment → `{shipment_id}` |
| `POST` | `/shipments/{id}/events` | ingest a tracking event (validated, queued in optimized mode) |
| `GET`  | `/shipments/{id}` | read shipment + recent events (cache + indexed SQLite) |
| `GET`  | `/healthz` | liveness + current mode |
| `GET`  | `/metrics` | ingest counters, cache stats, queue backlog |
| `POST` | `/admin/reset` | truncate tables + drain the stream (benchmark helper) |

Event validation: `event_type` must be one of `PICKUP, IN_TRANSIT, OUT_FOR_DELIVERY, DELIVERED,
EXCEPTION`; `location` is required. Invalid payloads are rejected with HTTP 422.

## Running it

### Prerequisites (Linux)
- Python 3.12, Java 21 (JDK), Maven, Redis. (`redis-server` must be running.)

### Setup
```bash
cd logistics-tracking-platform
make setup        # python venv + pip install, and mvn package for the Java worker
make redis        # start redis if it is not already running
```

### Live end-to-end demo
```bash
make demo         # starts ingest + worker, walks a shipment through the pipeline,
                  # shows validation rejection, retries, and the final cached read
```

### Reproduce the metrics
```bash
make bench            # Metric 1a, 1b, 2 and 3
make bench-pipeline   # Metric 1a + 2  (sync-inline vs async-offload architecture)
make bench-routing    # Metric 1b      (fifo vs tuned priority request routing)
make bench-read       # Metric 3       (pure SQLite read latency)
```

### Run a service by hand
```bash
make ingest INGEST_MODE=optimized   # FastAPI on :8080 (uvloop); use INGEST_MODE=baseline to compare
make worker                         # Java background-worker pool
```

| var | default | meaning |

| `INGEST_MODE` | `optimized` | `optimized` (async + queue) or `baseline` (sync inline) |
| `WORKER_THREADS` | `12` | background worker pool size |

| `TRANSIENT_FAIL_RATE` | `0.0` | injected transient failure probability per attempt |

| `SLA_MS` | `750` | event must be persisted within this window to meet SLA |
| `CACHE_ENABLED` | `true` | toggle the in-memory read cache |

## Defending it in an interview

- **"Show me the two services."** `ingest-service/` is Python/FastAPI; `tracking-worker/` is Java 21
  built with Maven. They communicate only through the Redis Streams queue and the shared SQLite store.
- **"Why is it faster?"** The baseline does the downstream call + DB write *inside* the request
  handler; the optimized path validates and hands the work to the background-worker pool over the
  queue, so the API returns in ~2 ms instead of ~22 ms (`make bench-pipeline` proves it live).
- **"What is the tuned request routing?"** `router.py` routes urgent events (`EXCEPTION`/`DELIVERED`)
  to a high-priority queue lane that workers drain before the normal lane; under a peak-load backlog
  that cuts urgent end-to-end response time from ~4.7 s (FIFO) to ~0.2 s (`make bench-routing`). The
  request tier is also tuned with the `uvloop` event loop and HTTP keep-alive, and can be scaled
  across CPUs with `SO_REUSEPORT` multi-worker uvicorn on Linux.
- **"How do you cut SLA breaches?"** Transient downstream failures are retried with backoff by the
  worker (and dead-lettered if they exhaust retries) instead of being dropped, so the breach rate
  falls from ~20% to ~0% under a 20% injected failure rate.
- **"How does indexing/caching help reads?"** `GET /shipments/{id}` filters events by `shipment_id`;
  the `idx_events_shipment` index turns a full scan into a seek, and the LRU cache serves hot
  shipments from memory — measured end to end in `benchmark/read_latency.py`.
- **"Is it correct under concurrency?"** SQLite runs in WAL mode; the upsert is idempotent (safe for
  at-least-once redelivery) and shipment status advances monotonically by event ingest time.
