- **Ingest Service** â€” Python / FastAPI (`ingest-service/`). Async REST handlers. In the optimized
  architecture the hot write path only **validates + publishes** to the message queue and returns
  immediately; reads are served from SQLite behind an in-memory LRU+TTL cache.
- **Tracking Worker** â€” Java 21 (`tracking-worker/`). A pool of consumer threads reads from a Redis

  transient failures with backoff, routes poison messages to a **dead-letter queue**, and persists
  results to SQLite.
- **Message queue** â€” Redis Streams with consumer groups (at-least-once delivery, acks, pending list).
- **Store** â€” a single SQLite database in WAL mode, shared concurrently by the Python reader and the
  Java writers, with secondary **indexes** on the hot query paths.

## REST API

| method | path | purpose |
| --- | --- | --- |
| `POST` | `/shipments` | create a shipment â†’ `{shipment_id}` |
| `POST` | `/shipments/{id}/events` | ingest a tracking event (validated, queued in optimized mode) |
| `GET`  | `/shipments/{id}` | read shipment + recent events (cache + indexed SQLite) |
| `GET`  | `/healthz` | liveness + current mode |
| `GET`  | `/metrics` | ingest counters, cache stats, queue backlog |
| `POST` | `/admin/reset` | truncate tables + drain the stream (benchmark helper) |

Event validation: `event_type` must be one of `PICKUP, IN_TRANSIT, OUT_FOR_DELIVERY, DELIVERED,
EXCEPTION`; `location` is required. Invalid payloads are rejected with HTTP 422.

## Running it

### Prerequisites (Linux)
- Python 3.12, Java 21 (JDK), Maven, Redis. (`redis-server` must be running.)

### Setup
```bash
cd logistics-tracking-platform
make setup        # python venv + pip install, and mvn package for the Java worker
make redis        # start redis if it is not already running
```

### Live end-to-end demo
```bash
make demo         # starts ingest + worker, walks a shipment through the pipeline,
                  # shows validation rejection, retries, and the final cached read
```

### Reproduce the metrics
```bash
make bench            # Metric 1, 2 and 3
make bench-pipeline   # only Metric 1 + 2  (live A/B of both architectures)
make bench-read       # only Metric 3      (pure SQLite read latency)
```

### Run a service by hand
```bash
make ingest INGEST_MODE=optimized   # FastAPI on :8080 (uvloop); use INGEST_MODE=baseline to compare
make worker                         # Java background-worker pool
```

## Configuration knobs (env vars)

| var | default | meaning |
| --- | --- | --- |
| `INGEST_MODE` | `optimized` | `optimized` (async + queue) or `baseline` (sync inline) |
| `WORKER_THREADS` | `12` | background worker pool size |
| `DOWNSTREAM_MS` | `20` | simulated downstream call latency |
| `TRANSIENT_FAIL_RATE` | `0.0` | injected transient failure probability per attempt |

| `SLA_MS` | `750` | event must be persisted within this window to meet SLA |
| `CACHE_ENABLED` | `true` | toggle the in-memory read cache |

## Defending it in an interview

- **"Show me the two services."** `ingest-service/` is Python/FastAPI; `tracking-worker/` is Java 21
  built with Maven. They communicate only through the Redis Streams queue and the shared SQLite store.
- **"Why is it faster?"** The baseline does the downstream call + DB write *inside* the request
  handler; the optimized path validates and hands the work to the background-worker pool over the
  queue, so the API returns in ~2 ms instead of ~22 ms (`make bench-pipeline` proves it live).
- **"How do you cut SLA breaches?"** Transient downstream failures are retried with backoff by the
  worker (and dead-lettered if they exhaust retries) instead of being dropped, so the breach rate
  falls from ~20% to ~0% under a 20% injected failure rate.
- **"How does indexing/caching help reads?"** `GET /shipments/{id}` filters events by `shipment_id`;
  the `idx_events_shipment` index turns a full scan into a seek, and the LRU cache serves hot
  shipments from memory â€” measured end to end in `benchmark/read_latency.py`.
- **"Is it correct under concurrency?"** SQLite runs in WAL mode; the upsert is idempotent (safe for
  at-least-once redelivery) and shipment status advances monotonically by event ingest time.
